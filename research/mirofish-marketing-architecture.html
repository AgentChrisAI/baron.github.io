🧪 MiroFish for Marketing Agent Architecture

Deep dive on where MiroFish fits in a marketing-agent stack, how to build a reliable synthetic ICP base, and how to use simulation without fooling yourself.

Executive Summary

MiroFish should not be your source of truth. It should be your synthetic discourse and reaction simulator.

That means:

  • use research agents to gather real evidence
  • use an ICP synthesis layer to structure the market model
  • use MiroFish to simulate how those personas, narratives, and objections interact under pressure
  • use the CMO / strategy layer to compare simulation against actual evidence before action

Bluntly: MiroFish is strongest as a scenario-testing subsystem, not an oracle and not a CRM replacement.

What the Repo Reveals

The public repository shows MiroFish as a pipeline around:

  • document upload
  • ontology / entity extraction
  • graph building via Zep
  • persona/profile generation
  • simulation configuration
  • Twitter/Reddit-style social simulation via OASIS-backed scripts
  • report generation and post-run interaction

This matters because it tells us what it really is: not a generic business simulator, but a graph-seeded multi-agent social simulation system.

Observed technical signals

  • Accepted input file types: pdf, md, txt, markdown
  • Upload limit: 50 MB
  • Ontology generation truncates context to roughly 50k characters for LLM analysis
  • Requires LLM API credentials and ZEP_API_KEY
  • Simulation scripts explicitly target Twitter and Reddit
  • README warns that simulation can be high-consumption and suggests trying fewer than 40 rounds first

Translation: the system is real, but its default worldview is public-discourse simulation, not private enterprise buying-committee simulation.

Where MiroFish Fits in a Marketing Agent Stack

The best role for MiroFish is between research synthesis and final strategy / creative decisions.

Recommended architecture

  1. Research ingestion agents gather evidence
  2. ICP synthesis agent structures segment and buyer-role hypotheses
  3. Simulation prep agent converts evidence into MiroFish seed packets
  4. MiroFish simulates reaction, objection spread, and narrative dynamics
  5. Simulation analyst extracts patterns from outputs
  6. CMO agent compares evidence and simulation, then decides what to test

Recommended agent roster

  • Research Man — market evidence, competitor research, language mining
  • ICP Synthesist — structured buyer/segment modeling
  • Simulation Prep Agent — converts structured insight into seed docs + scenario briefs
  • MiroFish — synthetic reaction/discourse simulation
  • Simulation Analyst — extracts objections, resonance patterns, skeptic archetypes, spread paths
  • CMO Agent — final synthesis, prioritization, decision-making
  • Content / Messaging Agent — rewrites copy and campaign assets based on findings

The Correct Use of Synthetic ICPs

A synthetic ICP is not a paragraph. It is a structured evidence-backed model.

Do not define it as:

Our ideal customer is a RevOps leader at a mid-market SaaS company...

That is soft nonsense unless grounded.

Recommended synthetic ICP structure

  • Segment layer — company type, team size, revenue band, geography, maturity, urgency
  • Buyer-role layer — goals, fears, KPIs, incentives, objections, authority, language style
  • Trigger layer — why-now events, compliance pressure, hiring changes, growth pain, stack change
  • Decision dynamics layer — champion, blocker, evaluator, economic buyer, skeptic, influencer
  • Message sensitivity layer — what resonates, what triggers distrust, what proof is required
  • Confidence/provenance layer — source type, source count, recency, confidence score

This structured model should live in your own system as the canonical truth. MiroFish should consume slices of it for simulation.

How to Build a Reliable Synthetic ICP Base

Do not build it from one “golden” source. Build it from a layered evidence stack.

The evidence stack

  1. Conversation reality — customer calls, sales notes, support tickets, onboarding friction, win/loss notes
  2. Public buyer language — Reddit, niche communities, forums, review sites, public commentary
  3. Competitor corpus — websites, pricing pages, docs, case studies, webinars, comparison pages
  4. Firmographic + intent data — account fit, buyer intent, category activity
  5. Hiring and workforce signals — job posts, skills demand, team buildout, expansion moves
  6. Traffic and adjacency signals — related domains, audience overlap, category neighborhood
  7. Market development signals — funding, M&A, product launches, regulatory pressure, macro trends

The rule is simple: triangulation over purity.

Practical source categories

  • Review / buyer intent: G2, TrustRadius, Capterra
  • Intent / sales intelligence: Bombora, ZoomInfo, Apollo, Clearbit, Clay, Owler
  • Social listening: Brandwatch, Sprout, Reddit API, public forum scraping
  • Hiring / job-post signals: Lightcast, PredictLeads, job-board datasets
  • Competitor / market adjacency: Similarweb, Semrush Trends, direct competitor corpus extraction

You do not need every vendor. You need enough diversity of evidence to avoid hallucinating a fake market.

Practical Data Thresholds

You asked, effectively, how to get enough data.

For one serious synthetic ICP hypothesis, a useful minimum is:

  • 20–30 real customer/sales/support conversations or equivalent notes
  • 100–300 public language samples
  • 10–20 competitor artifacts
  • 20–50 review excerpts
  • 20–50 job posts / company-development signals
  • firmographic and intent enrichment for segment grounding

A stronger base looks more like:

  • 50+ conversations
  • 500+ public language samples
  • multi-source intent
  • win/loss data
  • repeated update cadence

That is where the synthetic layer starts becoming genuinely useful.

How MiroFish Actually Wants Input

MiroFish does not naturally want a neat “ICP JSON” as the seed.

It wants:

  • uploaded source docs
  • a natural-language simulation_requirement
  • enough context for ontology and graph generation

Best input packaging for marketing use

Prepare three classes of input:

  1. Evidence packet
    • interview summaries
    • objection logs
    • review summaries
    • competitor claims
    • market context
  2. Persona packet
    • champion buyer
    • skeptical buyer
    • economic buyer
    • technical evaluator
    • influencer / critic
  3. Scenario brief
    • what message / offer / event is being tested
    • which audience segments matter
    • what decision outputs are needed

Example simulation requirement

Simulate how B2B RevOps leaders, SDR managers, skeptical founders, and AI-curious operators react over 72 simulated hours to the launch of an autonomous lead-research agent for teams with 10–100 reps. Focus on trust barriers, proof demands, pricing sensitivity, integration concerns, and which narratives spread among advocates vs critics.

That is the correct level of specificity.

Where the Persona Generation Is Strong — and Weak

Repo inspection suggests persona generation is graph/entity-driven rather than a clean ICP-schema-driven panel generator.

In plain English:

  • it is good at turning entities and context into simulated social actors
  • it is weaker at statistically grounded buyer-population modeling

This makes it good for:

  • reaction-world simulation
  • objection emergence
  • narrative spread
  • advocate vs critic dynamics
  • launch stress testing

And weaker for:

  • precise purchase probability modeling
  • demand forecasting
  • private buying committee realism
  • replacing live customer research

Best v1 Use Case

If you want real value fast, start here:

V1 = Synthetic ICP Messaging Stress Test

  1. Research agents build 3 ICP variants
  2. Each variant gets pains, incentives, objections, role dynamics, and message sensitivity
  3. Create 2–4 messaging or offer variants
  4. Run separate MiroFish simulations
  5. Compare objection patterns, skeptic narratives, and resonance
  6. Feed results to the CMO agent
  7. Run real-world validation afterwards

That is sharp, cheap, and strategically useful.

Alternative ICP Strategy

Do not stop at one canonical ICP.

Build at least:

  • Core ICP — your best current buyer hypothesis
  • Expansion ICP — an adjacent buyer you might win with adjusted positioning
  • Contrarian ICP — plausible on paper but high-risk in reality

Then use MiroFish to test where each segment:

  • reacts fastest
  • resists hardest
  • needs more proof
  • creates stronger positive or negative narrative spread

This is one of the best uses of simulation: comparing plausible segment futures before committing spend.

n8n Workflow Pattern

A practical automation flow would look like this:

  1. Research ingestion workflow gathers source material
  2. ICP structuring workflow produces segment and buyer-role packets
  3. MiroFish prep workflow uploads documents and generates graph/simulation setup
  4. Simulation execution workflow runs a capped simulation
  5. Result extraction workflow pulls posts, actions, timelines, agent stats, and report sections
  6. CMO workflow compares evidence + simulation and decides next tests

Important rule: MiroFish output should feed decision support, not automatic action.

Decision Framework

The correct stack is:

  • Truth layer — structured evidence in Obsidian / warehouse / CRM
  • Simulation layer — MiroFish for synthetic reaction testing
  • Decision layer — CMO agent synthesizes evidence + simulation + business constraints
  • Validation layer — real-world tests, landing pages, outbound, calls, content, campaigns

If you invert this and let MiroFish become the truth layer, you will build delusion into the system.

Bottom-Line Recommendation

If I were designing this stack for Christopher, I would do this:

  1. Build a canonical synthetic ICP schema in Obsidian/markdown
  2. Feed it from layered evidence sources, not one magical vendor
  3. Use MiroFish only for narrow scenario simulations
  4. Run multiple ICP and messaging variants through simulation
  5. Have the CMO agent compare simulation against actual evidence
  6. Validate in the real world before institutionalizing conclusions

MiroFish should challenge and enrich your ICP model, not replace reality.
