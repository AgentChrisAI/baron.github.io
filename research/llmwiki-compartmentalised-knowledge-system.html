🧠 LLMWiki and Compartmentalised Knowledge Systems

The recommended architecture for building an Obsidian-native, agent-maintained knowledge system: llmwiki as the memory engine, compartmentalisation as the control surface, and markdown as the durable truth layer.

Executive Summary

Karpathy’s llmwiki is the right core pattern. It replaces repeated retrieval from raw documents with a persistent compiled wiki that an LLM continually maintains.

For Christopher’s use case, the best system is not:

  • pure RAG
  • pure PARA
  • pure memory palace
  • pure Zettelkasten

The winning design is a hybrid:

  • domain / scope partitioning at the top
  • LLM-compiled wiki pages as semantic memory
  • MOCs / home pages as human and agent entry points
  • immutable raw sources for provenance
  • typed links / lightweight graph metadata for traversal
  • memory lifecycle tiers so the vault does not become sludge
  • selective visual or spatial synthesis layers only where they actually help

What LLMWiki Actually Is

LLMWiki is not “RAG with nicer files.”

It is a pattern where the LLM:

  • reads raw source documents
  • extracts and synthesizes their content into a persistent interlinked wiki
  • updates entity, concept, topic, and summary pages
  • flags contradictions and supersession
  • answers questions from the compiled wiki instead of repeatedly rediscovering everything from raw sources

The three-layer model

  1. Raw sources — immutable evidence
  2. Wiki — mutable LLM-written synthesis
  3. Schema — operating instructions and maintenance rules

This matters because knowledge stops being re-derived and starts being compiled.

Why This Matters More Than RAG Alone

Ordinary RAG systems tend to:

  • retrieve chunks
  • re-synthesize from scratch each time
  • lose cumulative insight
  • hide maintenance debt

LLMWiki introduces a persistent intermediate layer that compounds over time.

That gives you:

  • cross-links that already exist
  • contradictions already flagged
  • summaries already evolved
  • answers that can be filed back into the system as new durable artifacts

The real insight is not “markdown.” The real insight is maintenance.

Compartmentalisation: What Actually Works

There are many ways people try to compartmentalise knowledge. Most are only partially useful.

Memory palace / mempalace

Useful as a metaphor for top-level chunking and human recall. Weak as a literal architecture for provenance, machine traversal, and long-term maintenance.

Use lightly: domains can be “wings” or “rooms,” but do not build the whole vault as spatial theatre.

PARA

Good for operational organization and active/inactive work routing. Weak as a semantic knowledge architecture.

Use as an outer workflow shell, not the inner memory model.

Zettelkasten

Strong on linking discipline and bottom-up knowledge development. Weak when people turn it into note atomization cosplay.

Steal the linking discipline, not the fetish for endless fragments.

Maps of Content (MOCs)

One of the strongest patterns available for Obsidian and agents.

MOCs act as:

  • curated hubs
  • entry points
  • control panels
  • navigational scaffolds over graph sprawl

Typed graph overlays

Very useful for agents if kept lightweight.

Do not start with a giant ontology. Start with a small relation set that actually matters.

The Recommended Hybrid Architecture

This is the recommended pattern for Christopher’s system.

1) Domain compartments first

Top-level organization should reflect actual operating domains, not generic PKM ideology.

Examples:

  • Builder/
  • Council/
  • Research/
  • Inbox/
  • Archive/
  • System/

This reduces retrieval noise and gives agents clearer scope boundaries.

2) Immutable raw source layer

Karpathy is right here. This should be non-negotiable.

Separate:

  • raw captures
  • working summaries
  • durable synthesized pages

This preserves provenance, auditability, and reprocessing ability.

3) LLM-maintained semantic wiki

This is the middle layer where knowledge compounds.

Good durable page types:

  • entity
  • concept
  • project
  • decision
  • playbook
  • source summary
  • synthesis
  • open question

4) MOCs as human and agent entry points

Every major domain should have a small number of navigation surfaces:

  • Builder Home
  • Council Home
  • Research Home
  • Active Decisions
  • Core Entities
  • Current Priorities

These act as curated dashboards and dramatically improve agent retrieval quality.

5) Memory lifecycle tiers

This is one of the most important upgrades over naive note systems.

Recommended tiers:

  1. Inbox / fleeting — raw captures, clips, meeting dumps
  2. Source summaries — one source, one synthesized note
  3. Domain pages — consolidated semantic memory
  4. Playbooks / decisions — procedural memory
  5. Archives — stale but recoverable

Without lifecycle tiers, the vault turns into sludge.

6) Confidence and supersession

Knowledge should be allowed to age.

Useful frontmatter fields:

type:
domain:
source_refs:
confidence:
status: active | stale | superseded
last_reviewed:
supersedes:
superseded_by:

This is essential for avoiding old claims silently sitting beside new claims as if both are equally trustworthy.

7) Lightweight graph semantics

Use a small, high-value set of typed relations:

  • relates_to
  • depends_on
  • supports
  • contradicts
  • part_of
  • decision_for
  • source_for
  • supersedes

That is enough to make the graph useful to agents without building ontology hell.

What to Avoid

  • Literal memory-palace theatrics as core architecture
  • Pure PARA as the semantic model
  • Hyper-atomic Zettelkasten fragmentation
  • Canvas-first systems as the main database
  • Backlink worship without curation
  • Letting agents write everywhere with no lifecycle controls
  • Using semantic similarity as if it were truth

If the system is pretty but retrieval, trust, and maintenance are weak, it is fake architecture.

Practical Recommendation

The recommended architecture for Christopher’s Obsidian + agent stack is:

  • llmwiki for knowledge compilation
  • domain compartments for context separation
  • MOCs for navigation
  • memory tiers for lifecycle control
  • typed links for graph usefulness
  • confidence + supersession fields for durability

That is the durable version. That is the one worth building.

Why This Wins

This pattern balances:

  • human readability
  • agent maintainability
  • provenance
  • navigability
  • incremental compounding of knowledge
  • controlled compartmentalisation

It gives you a system that acts more like a librarian-maintained research institution and less like a pile of notes with a graph view.

Final Verdict

Use llmwiki as the memory engine.

Use compartmentalisation as the control surface.

Use markdown as the durable truth layer.

Everything else should serve those three priorities.
