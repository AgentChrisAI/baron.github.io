🏗️ Hybrid Vault Architecture

The concrete vault design for Christopher’s knowledge operating system: Raw, Restricted, and Main trust zones; client insight promotion; chronology; review queues; and agent access boundaries.

Executive Summary

This system should be built as a hybrid trust architecture with three layers:

  • Raw — immutable archive of original materials
  • Restricted — private and confidential compiled knowledge
  • Main — general operational brain for business, research, projects, and safe promoted insight

The key principle is simple:

raw confidential material stays protected, but high-value sanitized learning must still flow upward into the main vault.

That avoids two common failure modes:

  • everything is mixed together and unsafe
  • everything is siloed and useful insights disappear

Architecture Overview

Layer 1 — Raw

The raw archive is immutable and comprehensive.

It stores:

  • original files
  • exports
  • PDFs
  • docs
  • spreadsheets
  • images
  • newsletters
  • chat exports
  • notes app exports
  • anything else the system ingests

Raw exists for:

  • preservation
  • reprocessing
  • auditability
  • pipeline failure recovery

Normal agents should not use this as a working memory surface.

Layer 2 — Restricted

This is the controlled semantic zone.

It stores:

  • private journals
  • highly sensitive personal material
  • legal/financial-risk material
  • confidential client summaries
  • restricted source notes
  • restricted decisions and patterns

This layer is queryable, but only by high-trust agents or explicit user approval paths.

Layer 3 — Main

This is the main operational brain.

It stores:

  • business knowledge
  • research
  • projects
  • playbooks
  • decisions
  • entity pages
  • concept pages
  • MOCs
  • client-safe promoted insights

This is where most agents should work.

Trust Model

The system should behave like two primary graphs plus one archive:

Main Business Graph

Heavily linked.

Should include:

  • business notes
  • projects
  • research
  • clients (safe layer)
  • playbooks
  • ideas
  • decisions
  • general operational knowledge

Restricted Personal Graph

Also linked, but strongly restricted.

Should include:

  • private journals
  • personal reflections
  • private life patterns
  • high-sensitivity personal notes

Cross-graph policy

No default linking between personal and business.

Allowed mode: rare manual bridge notes only.

This matches the chosen model: C2.

Client Knowledge Promotion Model

This is the most important structural rule.

Client information should exist at three levels:

Level A — Raw client material

  • contracts
  • raw meeting notes
  • internal documents
  • exports
  • confidential artifacts

Location: Raw archive and/or restricted client zone.

Level B — Restricted client summaries

  • project summaries
  • issues
  • outcomes
  • patterns
  • lessons

Location: Restricted layer.

Level C — Sanitized transferable insight

  • generalized lessons
  • repeatable patterns
  • anti-patterns
  • domain playbooks
  • de-identified strategic insights

Location: Main vault.

Promotion policy

The chosen model is B3:

  • agents can propose promotion candidates
  • a higher-trust agent approves them
  • only sanitized and non-identifying insight reaches Main

This preserves business learning without leaking client confidentiality.

What Lives Where

Raw

Contains originals only. Never treated as the main working memory.

Staging

Contains pipeline artifacts:

  • OCR text
  • metadata
  • classification output
  • sensitivity scan results
  • dedup clusters

Restricted

Contains restricted compiled knowledge and review queues.

Main

Contains the agent-usable operating memory.

Chronology

Chronology is important, so the architecture must preserve time explicitly.

Every ingested item and durable page should track:

  • source date
  • ingest date
  • last reviewed date
  • era / timeline grouping where useful
  • supersession status

Main should include:

  • timelines/ for domain or project histories
  • decision timelines
  • idea evolution pages

Pipeline Flow

Primary ingestion flow

  1. file lands in Raw
  2. hash + manifest created
  3. text extraction / OCR / parsing runs
  4. metadata normalized
  5. sensitivity classification runs
  6. domain destination suggested
  7. source note created
  8. summary created
  9. entities/topics/dates extracted
  10. related pages proposed
  11. MOC updates proposed
  12. retrieval index updated
  13. review queue item created if necessary

Parallel dedup flow

Dedup should be a separate pipeline, not a blocker.

  1. compare hashes / metadata / text similarity
  2. cluster near-duplicates
  3. detect likely supersession
  4. detect recurring themes
  5. suggest merges, links, or archival decisions
  6. report to review layer

This matches the chosen design: dedup runs in parallel.

Agent Access Policy

Default rule

Agents should not read raw data by default.

General agents may access:

  • Main MOCs
  • Main concepts
  • Main entities
  • Main source notes
  • Main playbooks
  • Main decisions
  • promoted client-safe insight

Restricted agents may access:

  • restricted summaries
  • confidential flags
  • promotion candidates
  • private journals where explicitly permitted

Raw access

Only special-purpose ingestion, audit, or high-trust agents should touch Raw.

Daily Review System

The selected daily review priorities are:

  1. Confidential flags — risk reduction
  2. Research gaps — power upgrade
  3. Client insight promotion candidates — power upgrade
  4. Unresolved contradictions — conflicting-information risk reduction
  5. Proposed MOC updates — navigation upgrade

This is good. It means the system is not just passively storing memory; it is actively improving itself.

Note Classes for Main

Main should be built around a small, durable note taxonomy:

  • type: source
  • type: concept
  • type: entity
  • type: project
  • type: decision
  • type: playbook
  • type: synthesis
  • type: moc
  • type: timeline

Core Metadata

Every durable page should be able to answer:

  • what is this?
  • what domain is it in?
  • where did it come from?
  • how confident are we?
  • is it active, stale, or superseded?
  • what does it link to?

Why This Architecture Wins

  • Raw preserves everything
  • Restricted protects what must be protected
  • Main remains richly linked and useful
  • client insight is not lost
  • personal and business do not bleed together
  • agents operate mostly on safe compiled memory
  • the system supports chronology, dedup, and review

Failure Modes Avoided

  • flat-vault chaos
  • confidential leakage into general retrieval
  • knowledge silos that waste valuable client learning
  • raw-file dependency as the only source of insight
  • pipeline misses that permanently lose information

Final Recommendation

Build the system as:

  • Raw archive for preservation and recovery
  • Restricted layer for private and confidential semantic memory
  • Main vault for the linked operational brain
  • Promotion workflow for moving sanitized insight from Restricted into Main
  • Parallel dedup pipeline
  • daily review loop that upgrades the system over time

This is the right foundation for the next steps: actual schemas, note templates, and agent roles.
