Agent 1 Research Report: Spec-Kit Deep Dive

Date: 2026-03-23

Agent: Baron Von Claw (Subagent — Research)

Sources: github/spec-kit repo (live), spec-driven.md, extensions/README.md, presets/README.md, all template files, GitHub API tree

Summary

GitHub's spec-kit (aka specify-cli) is a mature, production-ready CLI toolkit that operationalises Spec-Driven Development — a methodology where specifications are the source of truth and code is the generated output. It ships with structured markdown templates, AI slash commands for every phase, and a growing extension/preset ecosystem. The core workflow (constitution → specify → clarify → plan → tasks → implement) maps surprisingly well to how Baron Von Claw already operates. Crucially: it's technology-agnostic in philosophy — there is nothing in the architecture that prevents non-code artifact targets (JSON workflows, Vapi configs), though the default templates and community examples lean heavily toward traditional software. The preset community catalog is currently empty (new system), while the extension catalog has 22+ community entries including Jira and Azure DevOps integrations.

This is what spec.md is based on — the output of /speckit.specify:

# Feature Specification: [FEATURE NAME]

**Feature Branch**: `[###-feature-name]`  
**Created**: [DATE]  
**Status**: Draft  
**Input**: User description: "$ARGUMENTS"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - [Brief Title] (Priority: P1)

[Describe this user journey in plain language]

**Why this priority**: [Explain the value...]
**Independent Test**: [How this can be tested independently]

**Acceptance Scenarios**:
1. **Given** [initial state], **When** [action], **Then** [expected outcome]

---

### Edge Cases
- What happens when [boundary condition]?
- How does system handle [error scenario]?

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: System MUST [specific capability]
- **FR-006**: System MUST authenticate users via [NEEDS CLARIFICATION: auth method?]

### Key Entities *(include if feature involves data)*
- **[Entity 1]**: [What it represents, key attributes without implementation]

## Success Criteria *(mandatory)*

### Measurable Outcomes
- **SC-001**: [Measurable metric]
- **SC-002**: [Measurable metric]

Critical design decision: The spec template explicitly forces:

  • ✅ Focus on WHAT users need and WHY
  • ❌ Avoid HOW to implement (no tech stack, APIs, code structure)
  • [NEEDS CLARIFICATION: ...] markers when anything is ambiguous (prevents LLM guessing)

Finding 4: Presets — How They Work, What Exists

Preset system: Presets customize how spec-kit works — they override templates and commands without adding new functionality. Resolution stack (highest to lowest priority):

.specify/templates/overrides/           ← Project-local one-off overrides
.specify/presets/<preset-id>/templates/ ← Installed presets
.specify/extensions/<ext-id>/templates/ ← Extension-provided templates
.specify/templates/                     ← Core templates

Key design features:

  • Presets are stackable with explicit priority ordering
  • Template resolution happens at runtime (not merged at install)
  • Command overrides applied at install time and written to agent directories
  • Multiple presets can coexist (e.g., enterprise-safe at priority 10, healthcare-compliance at priority 5)

Finding 5: Git/Branch Workflow Integration

The branching workflow is tightly integrated:

Branch naming:

  • Default: sequential numbering → 001-feature-name, 002-next-feature, 003-chat-system
  • Alternative: timestamp-based → 20260323-141523-feature-name (for distributed teams avoiding conflicts)

Configured at init:

specify init my-project --ai claude --branch-numbering sequential   # default
specify init my-project --ai claude --branch-numbering timestamp    # for teams

Non-git mode: If git is not desired, --no-git skips repo initialization. The SPECIFY_FEATURE env var allows working with a specific feature without git branches:

export SPECIFY_FEATURE=001-my-feature   # Override feature detection

Workflow lifecycle:

  1. /speckit.specify → triggers create-new-feature.sh → creates branch + spec directory
  2. Each feature lives on its own branch with its own specs/###-feature-name/ directory
  3. /speckit.plan → triggers setup-plan.sh → populates plan.md, research.md, contracts/
  4. /speckit.taskstoissues → creates GitHub Issues from tasks (requires GitHub MCP server)
  5. Normal git flow: commit spec changes alongside code, merge feature branch to main

The pattern mirrors how experienced engineering teams work — specs as PRs, code following from specs.

Finding 6: What's in spec-driven.md — Full Methodology

spec-driven.md is the authoritative philosophy document. Key sections:

The Power Inversion:

"Spec-Driven Development inverts the power structure. Specifications don't serve code — code serves specifications."

The core thesis: traditional specs were scaffolding discarded once "real work" began. SDD makes specs executable — they generate implementation rather than merely guiding it. "The lingua franca of development moves to a higher level, and code is the last-mile approach."

The Nine Constitutional Articles (from the default spec-kit constitution):

  1. Article I — Library-First: Every feature must begin as a standalone library
  2. Article II — CLI Interface Mandate: Every library must expose via CLI (text in/out, JSON support)
  3. Article III — Test-First Imperative: NON-NEGOTIABLE. Tests written → approved → confirmed failing → THEN implement
  4. Articles IV-VI: Integration testing, observability, versioning
  5. Article VII — Simplicity: Maximum 3 projects for initial implementation; justify any more
  6. Article VIII — Anti-Abstraction: Use framework directly; single model representation
  7. Article IX — Integration-First Testing: Real databases over mocks, actual service instances over stubs

Why SDD now (3 reasons given):

  1. AI capabilities crossed the threshold for reliable spec-to-code generation
  2. Software complexity grows exponentially — systematic alignment requires spec-first
  3. Pace of change demands pivoting as normal workflow, not exceptional event

Template constraints on LLMs (the real mechanism):

  • Specs explicitly instruct: ✅ Focus on WHAT, ❌ Avoid HOW
  • [NEEDS CLARIFICATION] markers prevent LLM assumption-making
  • Checklists act as "unit tests" for spec quality
  • Constitution compliance gates prevent over-engineering
  • "Phase -1 gates" for simplicity (≤3 projects?) and anti-abstraction (using framework directly?)
  • Test-first ordering embedded in task creation

SDD lifecycle phases:

  • 0-to-1 (Greenfield): Generate from scratch
  • Creative Exploration: Parallel implementations from same spec
  • Iterative Enhancement (Brownfield): Add features to existing codebases

Quality and validation commands:

/speckit.taskstoissuesConvert tasks to GitHub Issues (requires GitHub MCP server)

How it works technically:

Each .claude/commands/speckit.*.md file is a prompt template. When Claude Code runs /speckit.specify "Build a chat feature", it:

  1. Reads the command file
  2. Executes the associated bash/PowerShell script via {SCRIPT} placeholder
  3. Uses template files from .specify/templates/
  4. Creates the appropriate output files

The scripts handle git operations (branch creation, etc.) while Claude handles AI generation.

Note for Codex CLI: Different invocation — $speckit-specify (skills mode, not slash commands).

Finding 8: Can Spec-Kit Be Used for Non-Code Projects?

Short answer: Yes, philosophically. Mechanically — with adaptation.

What works out of the box:

  • The methodology is explicitly described as "not tied to specific technologies, programming languages, or frameworks"
  • Specs focus on WHAT, not HOW — which is equally valid for n8n workflow JSON or Vapi configs
  • The constitution and specify phases are purely natural language and technology-agnostic
  • The /speckit.plan phase can specify non-code targets — plan.md just captures your tech context

What requires adaptation:

The default templates assume code-first deliverables:

  • tasks-template.md references src/, tests/, code file paths
  • contracts/ assumes API specs (JSON schemas, REST endpoints)
  • The /speckit.implement command assumes an AI agent that writes code files

For n8n workflows specifically:

  • spec.md → Define the automation: user stories like "Given a lead form submission, When Typeform webhook fires, Then create contact in HubSpot and send welcome email"
  • plan.md → Specify: n8n workflow JSON structure, node types, credential names, error handling patterns
  • contracts/ → Define trigger schemas, API payloads, expected output format
  • tasks.md → Tasks like "T001: Create Webhook trigger node", "T002: Add HubSpot node with field mapping"
  • The AI agent (Claude Code) can generate n8n workflow JSON as deliverable

Precedent: The brownfield demos work with Go, Java, .NET — the methodology proved portable. Non-code artifacts are a natural extension. A custom preset could override spec-template.md to use n8n-specific terminology and structure.

The gap: There's no built-in "deploy" or "test" mechanism for non-code artifacts. /speckit.implement works by having Claude write code. For n8n JSON, you'd need the AI to produce the JSON file and either a manual import step or a custom extension that calls n8n's API to deploy.

Finding 10: Limitations — What Spec-Kit Does NOT Do Well

1. Preset ecosystem is immature (currently empty)

The community preset catalog has zero entries as of March 2026. Extensions are more developed. Anyone building presets is a first-mover.

2. Heavy code-software orientation in defaults

Default templates, examples, and community walkthroughs are 100% software code projects. Non-code artifact targets (n8n JSON, Vapi configs) require template customization. No built-in guidance for this use case.

3. No built-in feedback loop from runtime to spec

The spec-driven.md describes a vision where "production metrics and incidents update specifications." In practice, this is manual — there's no tooling to auto-update specs from production data.

4. Requires developer toolchain setup

Dependencies: Python 3.11+, uv (package manager), git, an AI agent CLI. Not trivially adopted by non-technical stakeholders. Installation is via uv tool install specify-cli --from git+....

5. The "implement" phase can be agent-unstable

From the Taskify demo, the walkthrough explicitly warns: "Claude Code might be over-eager and add components that you did not ask for." Human review is needed at multiple checkpoints. This is not fully autonomous.

6. Context pollution is a known problem

The Conduct Extension exists specifically to address context pollution from running all spec-kit phases in one session. Long sessions degrade AI quality — this is an inherent LLM limitation that spec-kit doesn't solve, just works around.

7. Sequential task execution is naive for complex parallel builds

The [P] parallelization markers in tasks.md identify parallel opportunities but the default implement command executes sequentially. Actual parallel execution requires the Fleet Orchestrator or Ralph Loop extensions.

8. No native state management between sessions

If a spec-kit session is interrupted mid-implementation, recovering requires the SDD Utilities extension (speckit-utils). The core toolkit has no built-in checkpointing.

9. Constitution is opt-in, not enforced

The architecture relies on the AI respecting the constitution. There's no hard enforcement mechanism. An AI agent can ignore architectural principles unless the template gates explicitly check them. This is a "soft" constraint.

10. No visual or UX layer

Spec-kit is entirely markdown files + CLI. There's no dashboard, no visual spec editor, no project management UI. The Project Status extension provides terminal-based visibility but it's text-only.
